Predictions & Anomaly Detection

How the app predicts next month's bill and flags high usage

The system uses statistical methods to analyze historical data, predict future trends, and detect anomalies (unusual spikes in energy usage).

const Analytics = {
  // Predict next month's bill based on the last 3 months
  predictNextBill(history) {
    if (history.length < 1) return 0;
    const recent = history.slice(-3); // get up to 3 most recent bills
    const sum = recent.reduce((acc, b) => acc + b.amount, 0);
    return Math.round(sum / recent.length * 1.05); // average + 5% expected increase
  },

  // Z-Score Anomaly Detection
  detectAnomaly(currentUsage, historyUnits) {
    const mean = historyUnits.reduce((a, b) => a + b, 0) / historyUnits.length;
    const variance = historyUnits.reduce((a, b) => a + Math.pow(b - mean, 2), 0) / historyUnits.length;
    const stdDev = Math.sqrt(variance);

    const zScore = (currentUsage - mean) / stdDev;
    return zScore > 2; // Flag as anomaly if usage is > 2 standard deviations above mean
  }
};

Faculty Q&A Cheat Sheet

Ready-made answers to common technical questions
Where is the database hosted?
For this prototype, we are using the browser's native localStorage API. It acts as a NoSQL document store, saving our users and bills as JSON arrays directly on the client.
How are you getting live sensor data?
Currently, the data is simulated using a JavaScript setInterval() function that generates randomized wattage values within realistic constraints for each appliance every 3.5 seconds. In a production environment, this would be replaced by WebSocket or MQTT subscriptions to real smart plugs.
How does the anomaly detection work?
We implemented a basic Z-score calculation. It looks at the historical mean and standard deviation of energy usage. If current usage exceeds the mean by more than 2 standard deviations, the system flags it as an anomaly.
Why use a Single Page Application (SPA)?
SPAs provide a seamless, app-like experience without page reloads. By handling DOM manipulation and routing purely in JavaScript, we reduce latency and can smoothly animate dashboard transitions.
How secure is the authentication?
This is a client-side prototype, so the authentication is simulated. Passwords are currently checked in plain text against the local JSON store. In a real-world deployment, we would use a backend server with hashed passwords (e.g., bcrypt) and JWT (JSON Web Tokens) for session management.
How did you generate the charts?
We used Chart.js, a popular open-source HTML5 canvas-based charting library. We dynamically inject the context (labels and datasets) based on which dashboard view the user is currently looking at.
What happens to the uploaded bill files?
The app intercepts the file input, reads the file name, and simulates an OCR extraction by generating randomized bill data. The actual file isn't uploaded to a server since we are running serverless.
What is the next scope for this project?
The next phase would involve connecting real ESP32 microcontrollers with current sensors to send live MQTT data, and migrating the storage to a cloud database like Firebase or MongoDB.
